CRUD + Base de Datos

Piensa en una app de tareas, como una agenda para organizarte en el colegio o en tu casa. Si la app solo te deja ver una pantalla bonita pero no guarda nada, no sirve de mucho. Lo importante es que puedas agregar tareas, verlas cuando quieras, cambiar una que esté mal escrita o eliminar la que ya terminaste.

Eso mismo pasa con aplicaciones de notas, tiendas online, redes sociales, sistemas de biblioteca o cualquier app que maneje información. Todas terminan haciendo lo mismo, aunque con nombres diferentes: CRUD.

💡 CRUD es el corazón de casi cualquier aplicación que maneje datos. Dominar estos conceptos te permitirá construir desde una simple lista de tareas hasta sistemas complejos de gestión.

Objetivos de aprendizaje

  • 🎯 Comprender el concepto de CRUD y cómo las cuatro operaciones básicas (Create, Read, Update, Delete) se aplican en APIs REST.
  • 💾 Explicar por qué necesitamos guardar datos de forma permanente usando bases de datos y las diferencias entre SQL y NoSQL.
  • 🔗 Mapear operaciones CRUD a métodos HTTP (GET, POST, PUT/PATCH, DELETE) y entender cuándo usar cada uno en endpoints REST.
  • 📊 Utilizar códigos de estado HTTP apropiados (200, 201, 400, 404, 500) para comunicar el resultado de cada operación.
  • ✅ Implementar validaciones de datos para proteger tu API de información inválida o maliciosa desde el cliente.

CRUD es un acrónimo que representa las cuatro operaciones básicas que se pueden realizar con datos:

  • Create (Crear): Agregar nuevos registros
  • Read (Leer): Consultar registros existentes
  • Update (Actualizar): Modificar registros existentes
  • Delete (Eliminar): Borrar registros

En una API REST, estas operaciones se mapean a métodos HTTP específicos. La combinación de método HTTP + ruta define qué acción se realiza:

Operación Método HTTP Endpoint Descripción
Create POST /tareas Crear una nueva tarea
Read (todas) GET /tareas Obtener todas las tareas
Read (una) GET /tareas/:id Obtener una tarea específica
Update PUT/PATCH /tareas/:id Actualizar una tarea
Delete DELETE /tareas/:id Eliminar una tarea

Lo importante no es memorizar cada línea, sino entender el principio: la ruta habla del recurso y el método habla de la acción.

Diferencia entre PUT y PATCH

  • PUT: Reemplaza completamente el recurso (debes enviar todos los campos)
  • PATCH: Actualiza parcialmente el recurso (solo envías los campos que cambiarán)

Para que todo esto sea útil, los datos deben guardarse en algún lugar permanente. Veamos las opciones:

❌ Guardar en memoria (variables)

let tareas = [{ id: 1, titulo: "Estudiar" }];

Problemas:

  • Al reiniciar el servidor, se pierden todos los datos
  • No se puede escalar a múltiples servidores
  • Limitado por la memoria RAM disponible

✅ Guardar en una Base de Datos

Una base de datos permite almacenar información de forma permanente, organizada y eficiente.

Tipos de Bases de Datos:

SQL (Relacionales)

Ejemplos: MySQL, PostgreSQL, SQLite

  • Datos en tablas con filas y columnas
  • Relaciones entre tablas
  • Lenguaje: SQL
  • Ideal para: Datos estructurados
NoSQL (No relacionales)

Ejemplos: MongoDB, Firebase, Redis

  • Documentos, colecciones, key-value
  • Estructura flexible
  • Lenguaje: JavaScript/JSON
  • Ideal para: Escalabilidad rápida

Los códigos de estado HTTP son números que el servidor envía al cliente para indicar el resultado de una petición. Son fundamentales para que el frontend sepa qué pasó.

Códigos más importantes para CRUD:

2xx - Éxito
  • 200 OK: Petición exitosa (GET, PUT, PATCH)
  • 201 Created: Recurso creado exitosamente (POST)
  • 204 No Content: Éxito sin contenido (DELETE)
4xx - Errores del cliente
  • 400 Bad Request: Datos inválidos o faltantes
  • 401 Unauthorized: No autenticado
  • 404 Not Found: Recurso no encontrado
5xx - Errores del servidor
  • 500 Internal Server Error: Error no manejado
  • 503 Service Unavailable: Servidor no disponible

❓ Pregunta rápida:

Incluso en proyectos pequeños, es crucial validar los datos que recibes. Nunca confíes en lo que envía el cliente.

Tipos de validaciones:

  1. Validación de presencia: ¿El campo existe?
  2. Validación de tipo: ¿Es del tipo correcto?
  3. Validación de formato: ¿Cumple con el formato esperado?
  4. Validación de rango: ¿Está dentro de límites aceptables?
  5. Validación de negocio: ¿Cumple con reglas específicas?

Para practicar CRUD sin complicarte con base de datos todavía, puedes empezar con un arreglo en memoria. No es "lo final" para un proyecto real, pero es perfecto para aprender cómo se conectan rutas, métodos y datos.

✅ Ventajas
  • No necesitas instalar BD
  • Enfócate en la lógica CRUD
  • Fácil de debuggear
  • Transición sencilla después
⚠️ Limitaciones
  • Datos se pierden al reiniciar
  • No escalable para producción
  • Sin persistencia real

📝 Actividad 1: Diseña tu CRUD

Elige una app: "lista de compras" o "videojuegos favoritos"

  1. Define el recurso principal (ej. compras o juegos)
  2. Escribe los endpoints CRUD
  3. Escribe 3 campos de tu tabla (ej. nombre, precio, fecha)

💡 Ejemplo de solución:

Recurso: /videojuegos

Endpoints CRUD:
- POST   /videojuegos       → Crear nuevo juego
- GET    /videojuegos       → Obtener todos los juegos
- GET    /videojuegos/:id   → Obtener un juego específico
- PATCH  /videojuegos/:id   → Actualizar un juego
- DELETE /videojuegos/:id   → Eliminar un juego

Campos:
- titulo (string)
- plataforma (string)
- completado (boolean)

✅ Actividad 3: Validaciones más reales

Mejora el endpoint POST /tareas:

  1. Si titulo tiene menos de 3 caracteres, responde 400 con un mensaje claro
  2. Si titulo viene con espacios al inicio o al final, recórtalo antes de guardar

💡 Código de ejemplo:

if (!titulo || titulo.trim().length < 3) {
  return res.status(400).json({
    success: false,
    error: "El título debe tener al menos 3 caracteres"
  });
}

const nuevaTarea = {

🔧 Actividad 4: Extensión del CRUD (nuevo campo)

Agrega el campo completada a tus tareas:

  1. Ajusta el POST /tareas para que cree { completada: false }
  2. Crea un endpoint PATCH /tareas/:id/completar que marque la tarea como completada

💡 Código de ejemplo:

app.patch("/tareas/:id/completar", (req, res) => {
  const id = Number(req.params.id);
  const tarea = tareas.find(t => t.id === id);
  
  if (!tarea) {
    return res.status(404).json({
      success: false,
      error: "Tarea no encontrada"
    });
  }
  
  tarea.completada = true;
  
  res.json({
    success: true,
    data: tarea
  });
});

🌐 Actividad 5: Mini-integración (cliente simple)

Crea un archivo HTML con un botón que haga fetch a GET /tareas y muestre el JSON en pantalla.

💡 Código de ejemplo:

<!DOCTYPE html>
<html lang="es">
<head>
  <meta charset="UTF-8">
  <title>Cliente CRUD</title>
</head>
<body>
  <h1>Mis Tareas</h1>
  <button onclick="obtenerTareas()">Obtener Tareas</button>
  <pre id="resultado"></pre>

  <script>
    async function obtenerTareas() {
      try {
        const response = await fetch('http://localhost:3000/tareas');
        const data = await response.json();
        document.getElementById('resultado').textContent = 
          JSON.stringify(data, null, 2);
      } catch (error) {
        console.error('Error:', error);
      }
    }
  </script>
</body>
</html>
